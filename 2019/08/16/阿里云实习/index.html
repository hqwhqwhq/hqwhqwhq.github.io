团队的整个框架

AHAS Demo −>Monkeyking −>ChaosBladeAHAS\ Demo \ -> Monkeyking \ -> ChaosBlade AHAS Demo −>Monkeyking −>ChaosBlade

简单介绍

ChaosBlade是一个故障注入工具

  1. 涉及场景：系统，应用服务，容器服务，云平台
  2. 不光有故障注入，还涉及运维检测，故障恢复等

Monkeyking是集团内部用于演练的一个平台

  1. 涉及演练流程编排，演练数据统计，权限控制等

AHAS是提供给外部用户使用的一个演练平台

AHAS Demo是提供给开源用户体验Chaosblade的一个分布式系统

AHAS Demo Docker 镜像

AHAS Demo涉及：base-provider, provider, consumer, zookeeper, redis, MySQL几个模块，分别将这些模块做成相应的镜像，然后使用Docker-compose来管理分布式系统的各个容器，制作成可以被开源用户一键部署的一个分布式系统

难点

  • 各个模块之间有相互的依赖关系，如何按照顺序启动不同的镜像？

解决方案

  • 在每个镜像中加入启动脚本，监测依赖镜像的端口是否启动，用来标志该镜像服务是否启动

Monkeyking平台

主要CRUD开发，没有特别地技术含量

  • 主要平台涉及到集团内部，公有云，专有云。专有云等涉及数据不能离线保存，因此需要提高接口的效率
  • 解决方案就是三个
    • 数据查询通过统一接口
    • 统一接口与DB之间加入缓存
    • 数据库查询使用多表连接查询

ChaosBlade功能开发

很开心贡献了一个开源社区代码，第一次PR

学习的方面

  • Go项目fork完了以后。本地最好先拉原项目（这样可以保证Go的路径正确），然后本地建立分支连接fork分支。
  • fork项目与远端更新的话，本地先拉取最新分支，再和fork对应的本地分支合并

CPU指定负载

  • 线程对应CPU。几个CPU就开几个线程，控制线程的占空比来控制总体的CPU占用率
  • 定时器反馈系统获取当前负载不断调整占空比

Memory指定负载

  • Linux下有文件系统为内存文件系统ramfs, tmpfs，通过向tmpfs文件系统的挂载点写入指定大小的文件来控制内存负载
    • 为啥不用ramfs文件系统
      • 这个文件系统需要先解挂载才能删除文件，并且不能看文件系统的内部文件系统大小
  • 以128M128M128M为最大一个文件大小，然后通过删除，增加文件来控制负载占比
  • 定时器反馈系统获取当前负载不断调整文件大小及个数
